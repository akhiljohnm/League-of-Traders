# Deriv API

> Comprehensive documentation for the Deriv API, including WebSocket and REST APIs for building trading applications.

**API Version:** This documents the Deriv API (the current production API).

## Overview

Deriv API is a WebSocket-based API that allows developers to:
- Access real-time market data and price feeds
- Execute trades programmatically
- Manage trading accounts and portfolios
- Build custom trading applications and bots

**Base REST URL:** `https://api.derivws.com`  
**Base WebSocket URL:** `wss://api.derivws.com/trading/v1/options/ws/{endpoint}` (authenticated via OTP — see REST API section)

---

## Quick Start

### 1. Authentication

Use the OAuth 2.0 Authorization Code flow with PKCE to obtain an access token (see **Authentication (OAuth 2.0)** section below). Then call `POST /trading/v1/options/accounts/{accountId}/otp` to get a one-time password (OTP). The response contains a ready-to-use WebSocket URL. For unauthenticated public data, connect directly to `wss://api.derivws.com/trading/v1/options/ws/public` without an OTP.

### 2. Your First API Call

```javascript
// For public data (no auth required):
const ws = new WebSocket('wss://api.derivws.com/trading/v1/options/ws/public');

// For authenticated trading, first obtain OTP via REST, then:
// const ws = new WebSocket(otpUrl); // URL returned by POST /trading/v1/options/accounts/{accountId}/otp

ws.onopen = () => {
  // Get server time (no auth required)
  ws.send(JSON.stringify({
    time: 1,
    req_id: 1
  }));
};

ws.onmessage = (event) => {
  console.log(JSON.parse(event.data));
};
```

## Authentication (OAuth 2.0)

Deriv uses the OAuth 2.0 Authorization Code flow with PKCE (Proof Key for Code Exchange) for authentication. This allows users to log in or sign up through Deriv's authorization server without sharing passwords with third-party apps.

**Authorization endpoint:** `https://auth.deriv.com/oauth2/auth`  
**Token endpoint:** `https://auth.deriv.com/oauth2/token`

### Prerequisites

- A registered OAuth2 client from Deriv with a `client_id` and a pre-registered `redirect_uri`
- HTTPS enabled on your redirect URL
- Your app must handle redirects, read the authorization code, and exchange it for tokens

### Flow Overview

1. **Generate PKCE** — Create a `code_verifier` (random 43-128 character string) and derive `code_challenge` = BASE64URL(SHA256(code_verifier)). Generate a random `state` for CSRF protection.
2. **Redirect to Deriv** — Send the user to Deriv's authorization endpoint with required parameters.
3. **User authenticates** — Deriv handles login/registration UI.
4. **Redirect back** — Deriv redirects to your `redirect_uri` with an authorization `code` and `state`.
5. **Verify state** — Confirm the returned `state` matches what you stored.
6. **Exchange code for token** — Your backend sends the `code` + `code_verifier` to the token endpoint and receives an `access_token`.
7. **Use the token** — Make authenticated API calls with the Bearer token.

### PKCE Parameters

| Term | Description |
|------|-------------|
| `code_verifier` | A cryptographically random string (43-128 characters) generated by your app |
| `code_challenge` | `BASE64URL(SHA256(code_verifier))` — sent with the authorization request |
| `code_challenge_method` | Always `S256` (SHA-256) |

### PKCE Generation Example (JavaScript)

```javascript
// 1. Generate a random code_verifier
const array = crypto.getRandomValues(new Uint8Array(64));
const codeVerifier = Array.from(array)
  .map(v => 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-._~'[v % 66])
  .join('');

// 2. Derive the code_challenge
const hash = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(codeVerifier));
const codeChallenge = btoa(String.fromCharCode(...new Uint8Array(hash)))
  .replace(/\+/g, '-')
  .replace(/\//g, '_')
  .replace(/=+$/, '');

// 3. Generate a random state for CSRF protection
const state = crypto.getRandomValues(new Uint8Array(16))
  .reduce((s, b) => s + b.toString(16).padStart(2, '0'), '');

// 4. Store before redirecting (sessionStorage survives the redirect)
sessionStorage.setItem('pkce_code_verifier', codeVerifier);
sessionStorage.setItem('oauth_state', state);
```

### Login

Redirect users to the authorization endpoint with these parameters:

| Parameter | Required | Value / Description |
|-----------|----------|---------------------|
| `response_type` | Yes | `code` |
| `client_id` | Yes | Your registered OAuth2 application ID |
| `redirect_uri` | Yes | Must exactly match the URI registered with Deriv |
| `scope` | Yes | `trade` or `admin` |
| `state` | Yes | Random string for CSRF protection — generate fresh for each request |
| `code_challenge` | Yes | BASE64URL(SHA256(code_verifier)) |
| `code_challenge_method` | Yes | `S256` |

**Login URL example:**

```
https://auth.deriv.com/oauth2/auth?
  response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &scope=trade
  &state=RANDOM_STATE
  &code_challenge=PKCE_CHALLENGE
  &code_challenge_method=S256
```

### Sign Up

Sign up uses the same base URL and parameters as login, plus additional parameters that tell Deriv to show the registration form and attribute the signup to your partner account:

| Parameter | Required | Description |
|-----------|----------|-------------|
| `prompt` | Yes | Set to `registration` to show the signup form instead of login |
| `sidc` | No | Your session ID (GUID) from the Deriv partner dashboard, used for tracking |
| `utm_campaign` | No | Campaign name for analytics (e.g. `summer2026`, `homepage`) |
| `utm_medium` | No | Set to `affiliate` for partner integrations |
| `utm_source` | No | Your affiliate ID from the Deriv partner dashboard (e.g. `CU303219`) |

**Sign Up URL example:**

```
https://auth.deriv.com/oauth2/auth?
  response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &scope=trade
  &state=RANDOM_STATE
  &code_challenge=PKCE_CHALLENGE
  &code_challenge_method=S256
  &prompt=registration
  &sidc=YOUR_SESSION_GUID
  &utm_campaign=YOUR_CAMPAIGN
  &utm_medium=affiliate
  &utm_source=YOUR_AFFILIATE_ID
```

### Callback Handling

After authentication, Deriv redirects to your `redirect_uri`:

**Success:** `https://yourapp.com/callback?code=AUTHORIZATION_CODE&state=RANDOM_STATE`  
**Error:** `https://yourapp.com/callback?error=access_denied&error_description=User+cancelled`

Your app must:
1. Verify the `state` matches the value stored before the redirect
2. Extract the authorization `code` from the URL parameters

The authorization code is single-use and expires quickly — exchange it immediately.

### Token Exchange

Make a POST request from your **backend** (never from the browser):

```http
POST https://auth.deriv.com/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&client_id=YOUR_CLIENT_ID
&code=AUTH_CODE_FROM_CALLBACK
&code_verifier=YOUR_ORIGINAL_CODE_VERIFIER
&redirect_uri=https://yourapp.com/callback
```

**cURL example:**

```bash
curl -X POST https://auth.deriv.com/oauth2/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "code=AUTH_CODE" \
  -d "code_verifier=YOUR_CODE_VERIFIER" \
  -d "redirect_uri=https://yourapp.com/callback"
```

**Token response:**

```json
{
  "access_token": "ory_at_...",
  "expires_in": 3600,
  "token_type": "Bearer"
}
```

### Using the Access Token

Include the token as a Bearer token in the `Authorization` header for all API calls:

```http
Authorization: Bearer YOUR_ACCESS_TOKEN
```

**Example — list accounts:**

```bash
curl -X GET "https://api.derivws.com/trading/v1/options/accounts" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

### OAuth Troubleshooting

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| State mismatch error | `state` in callback doesn't match stored value | Store `state` in `sessionStorage` before redirecting |
| `invalid_grant` on token exchange | `code_verifier` doesn't match the challenge, or code expired | Send the original `code_verifier`, not a new one; exchange immediately |
| Redirect URI mismatch | URL doesn't exactly match registration | Check trailing slashes, http vs https, port numbers |
| `invalid_client` | Wrong `client_id` | Verify credentials from the Deriv dashboard |
| Login form shows instead of signup | Missing `prompt=registration` | Add `prompt=registration` to the authorization URL |
| Signup not tracked to partner | Missing or wrong UTM parameters | Verify `sidc`, `utm_source`, `utm_medium`, and `utm_campaign` are correct |

### Trading Operations

#### proposal

Get a price proposal for a specific contract. Returns pricing information before buying.

**Request:**
```json
{
  "proposal": 1,
  "amount": 100,
  "basis": "stake",
  "contract_type": "CALL",
  "currency": "USD",
  "duration": 5,
  "duration_unit": "t",
  "underlying_symbol": "1HZ100V",
  "subscribe": 1,
  "req_id": 1
}
```

**Parameters:**
- `proposal` (integer, required): Must be 1
- `contract_type` (string, required): Contract type (CALL, PUT, DIGITEVEN, etc.)
- `currency` (string, required): Account currency
- `underlying_symbol` (string, required): Trading symbol
- `amount` (number, optional): Proposed contract payout or stake
- `basis` (string, optional): "stake" or "payout"
- `duration` (integer, optional): Contract duration (required if date_expiry not set)
- `duration_unit` (string, optional): "s" (seconds), "m" (minutes), "h" (hours), "d" (days), "t" (ticks)
- `date_expiry` (integer, optional): Epoch expiry time (alternative to duration)
- `barrier` (string, optional): Contract barrier
- `barrier2` (string, optional): Second barrier for range contracts
- `growth_rate` (number, optional): Growth rate for accumulator (ACCU) contracts
- `cancellation` (string, optional): Cancellation duration option (only for MULTUP and MULTDOWN contracts)
- `limit_order` (object, optional): Add stop_loss and/or take_profit orders (for MULTUP, MULTDOWN, and ACCU contracts)
  - `stop_loss` (number): Auto-close at specified loss
  - `take_profit` (number): Auto-close at specified profit
- `multiplier` (number, optional): Multiplier for non-binary options (e.g., lookbacks)
- `payout_per_point` (number, optional): Payout per point; barrier auto-calculated from this value
- `selected_tick` (integer, optional): Predicted tick with highest/lowest value (for TICKHIGH and TICKLOW)
- `subscribe` (integer, optional): 1 to subscribe to price updates

**Response:**
```json
{
  "proposal": {
    "id": "abc123xyz",
    "ask_price": 10.50,
    "date_start": 1234567890,
    "longcode": "Win payout if Volatility 100 Index is strictly higher than entry spot at 5 ticks after contract start time.",
    "payout": 19.90,
    "spot": 5123.44,
    "spot_time": 1234567890
  },
  "msg_type": "proposal",
  "req_id": 1
}
```

**Key response fields:**
- `id` (string, required): Proposal ID — pass to `buy` to purchase. Also used as subscription ID for `forget`.
- `ask_price` (number): The price to buy the contract
- `payout` (number): The potential payout
- `spot` (number): Current spot price of the underlying
- `spot_time` (integer): Epoch time of spot
- `date_start` (integer): Contract start time
- `date_expiry` (integer): Contract expiry time
- `longcode` (string): Human-readable contract description
- `cancellation` (object, optional): Cancellation option details (ask_price, date_expiry) for MULTUP/MULTDOWN
- `limit_order` (object, optional): Stop loss/take profit/stop out order details
- `contract_details` (object, optional): Extra info for Accumulators (tick_size_barrier, maximum_payout, maximum_ticks, ticks_stayed_in) and barrier details
- `validation_params` (object, optional): Min/max constraints for stake, payout, take_profit, stop_loss

**Authentication:** Not required (but needed for personalized pricing)

## Error Handling

All API responses may include an error object:

```json
{
  "error": {
    "code": "AuthorizationRequired",
    "message": "Please log in."
  },
  "msg_type": "authorize",
  "req_id": 1
}
```

### Common Error Codes

- `AuthorizationRequired` - Need to authorize first
- `InvalidToken` - Token is invalid or expired
- `RateLimit` - Too many requests
- `InputValidationFailed` - Invalid request parameters
- `ContractNotFound` - Contract doesn't exist
- `InsufficientBalance` - Not enough funds
- `ValidationError` - Field validation failed
- `NotFound` - Resource not found
- `Unauthorized` - Authentication failed
- `InternalError` - Server error occurred

## WebSocket Best Practices

1. **Keep connections alive**: Send periodic ping messages every 30 seconds
2. **Handle reconnection**: Implement exponential backoff for reconnects
3. **Use req_id**: Always include req_id to match responses to requests
4. **Manage subscriptions**: Use forget/forget_all to clean up subscriptions when done
5. **Error handling**: Always check for error responses before processing data
6. **Connection limits**: Maximum 100 active subscriptions per connection
7. **Request throttling**: Limit to 100 requests per second per connection

### Complete Trading Flow Example

```javascript
// Step 1: Obtain OTP via REST API (requires OAuth2 token)
const otpResponse = await fetch(
  'https://api.derivws.com/trading/v1/options/accounts/YOUR_ACCOUNT_ID/otp',
  {
    method: 'POST',
    headers: {
      'Deriv-App-ID': 'YOUR_APP_ID',
      'Authorization': 'Bearer YOUR_OAUTH_TOKEN'
    }
  }
);
const { data: otpData } = await otpResponse.json();

// Step 2: Open authenticated WebSocket using the OTP URL
// otpData.url === "wss://api.derivws.com/trading/v1/options/ws/demo?otp=..."
const ws = new WebSocket(otpData.url);

ws.onopen = () => {
  // Connection is authenticated via OTP — no authorize step needed
  // Step 3: Get proposal
  ws.send(JSON.stringify({
    proposal: 1,
    amount: 10,
    basis: "stake",
    contract_type: "CALL",
    currency: "USD",
    duration: 5,
    duration_unit: "t",
    underlying_symbol: "1HZ100V",
    req_id: 1
  }));
};

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  
  if (data.msg_type === 'proposal') {
    // Step 4: Buy contract
    ws.send(JSON.stringify({
      buy: data.proposal.id,
      price: data.proposal.ask_price,
      req_id: 2
    }));
  }
  
  if (data.msg_type === 'buy') {
    console.log('Contract purchased:', data.buy.contract_id);
    
    // Step 5: Subscribe to contract updates
    ws.send(JSON.stringify({
      proposal_open_contract: 1,
      contract_id: data.buy.contract_id,
      subscribe: 1,
      req_id: 3
    }));
  }
  
  if (data.msg_type === 'proposal_open_contract') {
    console.log('Contract status:', data.proposal_open_contract);
    
    // Optional: Sell when profitable
    if (data.proposal_open_contract.profit > 5) {
      ws.send(JSON.stringify({
        sell: data.proposal_open_contract.contract_id,
        price: data.proposal_open_contract.bid_price,
        req_id: 4
      }));
    }
  }
};

ws.onerror = (error) => {
  console.error('WebSocket error:', error);
};

ws.onclose = () => {
  console.log('WebSocket connection closed');
  // Implement reconnection logic here
};
```
